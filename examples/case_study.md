# Exploratory real-model case study (NOT validation)

> **Scope and honesty note.** This is an *exploratory* case study, not a
> validation or a benchmark. Everything below comes from tiny CPU configurations
> on synthetic toy data with a single seed per model and no hyperparameter
> search. Each model is described **on its own terms** — the observed timescale
> structure of its memory trajectory. There is **no cross-model comparison, no
> ranking, and no quality comparison between models**, nor any claim that a
> recovered timescale equals an injected one. Real-model validation is deferred
> to a future release.

All numbers here are machine-generated by
[`run_case_study.py`](run_case_study.py) and mirror
[`../results/realmodel_v0.2.json`](../results/realmodel_v0.2.json) (the
provenance file, env-stamped). Re-run with:

```bash
pip install -e ".[torch]" && pip install "titans-pytorch==0.5.3"
python examples/run_case_study.py
```

## What the study does

Two small memory models are trained on the same toy task — **next-step
prediction** of an 8-channel signal whose target mixes a fast (τ≈5) and a slow
(τ≈50) timescale. For each model we record a memory trajectory on a *held-out*
probe ensemble (disjoint seeds) **before** and **after** training, and run
`chronospect.analyze` on each.

The question is deliberately narrow: **does the instrument respond to learning?**
An untrained model's forward pass carries no task structure, so a change in the
recovered spectrum between *before* and *after* is the evidence that the readings
reflect learning-induced structure rather than an artefact of the probe. The
script enforces this mechanically — it asserts `training_steps > 0` **and** that
the recovered spectrum is not identical before vs after, so a forward-only
(hollow) study cannot pass.

The injected timescales (5, 50) describe the *target signal*, not a ground truth
the model's memory is expected to reproduce. We report what the instrument
observes, nothing more.

## vanilla GRU (`nn.GRU`)

* Memory trajectory: top-layer GRU hidden state per step (`record_rnn_states`).
* Training: 300 steps, MSE loss **1.0365 → 0.1841** (the model learned the task).

| | dominant timescales | effective # timescales | verdict |
|---|---|---|---|
| before training | [5.2, 34.7] | 2.90 | multi-timescale; non-stationary (aging) |
| after training  | [3.6, 34.4] | 2.98 | multi-timescale; non-stationary (aging) |

The recovered spectrum shifts with training (the fast peak moves from ≈5.2 to
≈3.6) while the verdict label stays the same. A randomly-initialised recurrent
net already carries multi-timescale state, so this is an observation about *how*
training reshapes the spectrum, not about the spectrum appearing from nothing.

## Titans `NeuralMemory` (titans-pytorch 0.5.3)

* Memory trajectory: per-step memory readout (`record_titans_memory`). We record
  the readout (the tensor the memory returns when queried) rather than the raw
  fast-weights, because a flattened fast-weight snapshot has dimension ≫ the
  number of chunks, which makes the capacity estimator's whitening singular; the
  readout is the low-dimensional *observable* memory signal and still reflects
  the test-time-updated memory state. See `record_titans_memory`'s docstring.
* Training: 30 steps, MSE loss **16.2318 → 0.9048** (the model learned the task).

| | dominant timescales | effective # timescales | verdict |
|---|---|---|---|
| before training | [49.8] | 2.00 | effectively single-speed; approximately stationary |
| after training  | [1.2, 27.5] | 2.15 | multi-timescale; approximately stationary |

Here the recovered structure changes more visibly: before training the readout
resolves to a single dominant timescale, and after training the instrument
resolves more than one. We report this as an observation of the readout's
timescale structure under this specific toy setup — not as evidence that the
architecture "is" multi-timescale in general.

## Limitations (read before quoting any number)

* **Exploratory, single config.** One tiny CPU model per architecture, one seed,
  one toy task, one probe. Not a benchmark; not a validation.
* **No cross-model claim.** The two sections are independent observations. The
  numbers are not comparable across models and imply no ordering between them.
* **Synthetic data.** The probe and training signals are synthetic AR mixtures
  (`chronospect.synthetic`), chosen because their timescales are known; they are
  not natural-language or real task data.
* **Calibration caveat carries over.** Recovered long timescales are
  shrinkage-affected estimates (see the calibration section of the README); read
  them as lower bounds, not exact values.
* **HOPE / Nested-Learning deferred.** The only public implementation is an
  unofficial repo with API-drift risk; it is deferred to a future release rather
  than included here. The script records this skip explicitly (it never silently
  drops a model).
